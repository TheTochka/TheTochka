## 1. Цели и принципы

### 1.1 Бизнес-цели бэкенда

- Принимать действия от мобильного клиента, которые **нельзя доверять клиенту** (секреты, оплата, создание подписок у провайдера).
- Хранить **заказы** и **состояние выдачи** подписки.
- Интегрироваться с **платёжным провайдером** (например platega.io или аналог) через **webhook**.
- Вызывать **бекенды партнёров** по **единому контракту** (Partner API) или через **адаптер**.
- Отправлять **письма** (Resend и др.) **только с сервера**, без утечки API-ключей в приложение.

### 1.2 Инженерные принципы (понятность, скорость, простота)

| Принцип | Что это значит на практике |
|--------|----------------------------|
| **Понятность** | Один входной документ (этот план + OpenAPI для HTTP API). Имена статусов заказа и событий — в одном глоссарии. |
| **Скорость** | Горячий путь: webhook → постановка в очередь → ответ 200 быстро. Тяжёлое — асинхронно. Кэш каталога на клиенте; бэк не дублирует лишнее. |
| **Лёгкость** | Минимум сервисов на старте: один API + одна БД заказов + очередь или встроенный worker. Не вводить Kafka «на будущее», пока нет нагрузки. |
| **Надёжность** | Идемпотентность webhook и job провижнинга, ретраи с backoff, явные таймауты к Partner API. |

## 2. Что уже есть в проекте (инвентаризация)

### 2.0 Текущий поток (VPS proxy) — актуально для репозитория

В **продакшен-режиме** каталог, отзывы, чат поддержки и синхронизация настроек/избранного идут через **VPS API** ([backend/vps_api](../backend/vps_api/)): клиент передаёт **Firebase ID token** (`Authorization: Bearer …`), сервер использует **Firebase Admin SDK** для Firestore. **Resend** и прочие серверные ключи — только на VPS (и опционально в документе `_config/runtime_secrets`). Клиент дополнительно использует **Firebase Auth** и **Firebase Storage** (вложения чата `support_chats/{uid}/…`).

Подробности: [FIREBASE_SERVER_SETUP.md](FIREBASE_SERVER_SETUP.md), [VPS_MIGRATION.md](VPS_MIGRATION.md), [backend/vps_api/README.md](../backend/vps_api/README.md).

### 2.1 Клиент (Flutter)

- **[lib/config/app_config.dart](../lib/config/app_config.dart):** `API_BASE_URL`, `USE_VPS_API_PROXY` (по умолчанию прокси включён), `USE_DIRECT_FIRESTORE` (только отладка), `QUOTE_REQUEST_PATH` (по умолчанию `/api/v1/quote-request`), `APP_REQUEST_SECRET` для заголовка `X-App-Secret` на запрос КП (тот же секрет в env/Firestore на сервере). **Ключей Resend в клиенте нет.**
- **[lib/services/quote_request_service.dart](../lib/services/quote_request_service.dart):** запрос КП — **POST на VPS** (Resend вызывается только на сервере).
- **[lib/services/api_client.dart](../lib/services/api_client.dart):** HTTP GET/POST/PUT JSON к базовому URL VPS.

### 2.2 Основной API на сервере (VPS)

- **[backend/vps_api/](../backend/vps_api/):** Node.js (`server.mjs`), эндпоинты `/api/v1/…` (каталог, отзывы, чат, sync, quote-request), systemd + Caddy — см. README и [VPS_MIGRATION.md](VPS_MIGRATION.md).

### 2.3 Пример / legacy (только Resend)

- **[backend/resend_quote/](../backend/resend_quote/):** минимальный отдельный сервис «POST → письмо через Resend» для экспериментов и документации; **не** основной путь приложения.

### 2.4 Firebase

- **`firestore.rules`**: правила доступа (каталог и связанные данные — по модели «клиент через VPS или dev-режим»).
- **`functions/`**: Cloud Function **`syncReviewAggregates`** — пересчёт `rating` / `reviewCount` при изменении отзывов (нужен Blaze). README описывает деплой и backfill.

### 2.5 Пробелы относительно целевой модели «маркетплейс + оплата + Partner API»

- Нет централизованной сущности **Order** на бэке.
- Нет **webhook оплаты** и **очереди провижнинга**.
- Нет спецификации **Partner API** в репозитории (как OpenAPI).
- Релизные сборки нужно проверять на отсутствие секретов в **`--dart-define`** и в коде (см. фазу 0).

## 3. Целевая архитектура (логические компоненты)

```
[Flutter app]
    │  HTTPS (JSON, опционально X-App-Secret / Firebase Auth token)
    ▼
[API Gateway / App Server]  ← один процесс на старте (Node, Bun, Go — на выбор команды)
    │
    ├── Модуль: публичные/полупубличные эндпоинты (КП, здоровье, версия)
    ├── Модуль: заказы (создание намерения покупки, статус)
    ├── Модуль: webhook оплаты (подпись, идемпотентность)
    ├── Модуль: провижнинг (вызов Partner API провайдера)
    ├── Модуль: почта (Resend)
    └── Модуль: админ/операции (позже; может быть за VPN + отдельная auth)

[Хранилище заказов и секретов]  ← Firestore / PostgreSQL / оба (см. раздел 4)

[Очередь задач]  ← Cloud Tasks / BullMQ+Redis / встроенная таблица «jobs» на первом этапе

[Partner Adapter или API провайдера]  ← внешняя система, контракт фиксируете вы
```

## 4. Границы ответственности: Firebase vs отдельный API-сервер

### 4.1 Рекомендуемое разделение (практично для роста)

| Задача | Где лучше |
|--------|-----------|
| Каталог провайдеров/тарифов, отзывы, чат в приложении | **Firestore** (уже используется) |
| Агрегаты отзывов | **Cloud Function** (уже есть) или клиент |
| Секреты Resend, токены Partner, ключи оплаты | **Только сервер** (Secret Manager / env на хостинге) |
| Заказы, платежи, провижнинг, аудит | **API-сервер + БД**, устойчивая к гонкам и удобная для отчётов |

### 4.2 Упрощённый старт (MVP)

- Один сервис **`backend/api`** (расширение текущего `vps_api` или новый каталог).
- Заказы в **PostgreSQL** (если нужны отчёты и JOIN) или в **Firestore** в коллекции `orders` с жёсткими правилами «писать только через Admin SDK на сервере».

Выбор зафиксировать в **ADR** (короткая запись: «Почему Postgres / почему Firestore для заказов»).

## 5. Фаза 0 — стабилизация и безопасность (до новых фич)

**Цель:** продакшен-гигиена без изменения пользовательских сценариев.

1. Убедиться, что **в релизной сборке** в клиент **не попадают** API-ключи почты и партнёров: проверить **`--dart-define`**, [lib/config/app_config.dart](../lib/config/app_config.dart) и прочие константы; **Resend только на VPS** (текущая архитектура это соблюдает — не регрессировать).
2. Задеплоить **VPS API** на HTTPS-домен; в приложении задать `API_BASE_URL`, при необходимости `APP_REQUEST_SECRET`.
3. Зафиксировать в README **чеклист релиза** (переменные окружения, health URL).
4. Проверить **firestore.rules** на соответствие реальным экранам (отзывы, чат, Storage).
5. Документировать **регион** Cloud Functions и лимиты Blaze для `syncReviewAggregates`.

**Критерий готовности фазы 0:** запрос КП с реального устройства идёт только через ваш сервер; секреты не в бинарнике.

## 6. Фаза 1 — единый HTTP API и договор с клиентом

**Цель:** один стиль URL, ошибок, версионирования.

1. Ввести префикс **`/api/v1/`** для всех новых маршрутов.
2. Стандартизировать ответ об ошибке:

```json
{
  "error": {
    "code": "invalid_email",
    "message": "Человекочитаемое пояснение",
    "request_id": "uuid"
  }
}
```

3. На каждый запрос — **`X-Request-Id`** (принимать от клиента или генерировать).
4. Публикация **OpenAPI 3.1** (YAML) в репозитории, например `docs/openapi/v1.yaml`.
5. Перенести существующий **`POST /api/quote-request`** в спецификацию как **`POST /api/v1/quote-request`** (старый путь оставить как редирект или deprecated на срок миграции).

**Критерий готовности:** фронт может генерировать типы из OpenAPI (опционально); новый разработчик понимает контракт без чтения исходников сервера.

## 7. Фаза 2 — заказы (Order) и статусная машина

**Цель:** любая покупка — записанный заказ с явным состоянием.

### 7.1 Минимальные статусы

| Статус | Смысл |
|--------|--------|
| `draft` | Черновик (опционально) |
| `pending_payment` | Создан заказ, ждём оплату |
| `paid` | Оплата подтверждена webhook |
| `provisioning` | Идёт вызов Partner API |
| `active` | Подписка выдана, данные доставлены пользователю |
| `failed` | Ошибка провижнинга или фатальная ошибка |
| `cancelled` | Отмена до оплаты или по политике |

### 7.2 Правила переходов

- Разрешённые переходы описать **таблицей** в коде (state machine), не «if-лапша» без документа.
- Любой переход логировать: `order_id`, `from`, `to`, `actor` (system / admin / webhook).

### 7.3 API для клиента (черновик)

- `POST /api/v1/orders` — создать заказ (провайдер, тариф, email, идентификатор пользователя Firebase optional).
- `GET /api/v1/orders/{id}` — статус (без секретов конфига в открытом виде; только маски / флаги «готово»).

**Критерий готовности:** можно пройти сценарий «создал заказ → вижу статус» без платежа (mock).

## 8. Фаза 3 — оплата (webhook) и идемпотентность

**Цель:** деньги не теряются при повторной доставке webhook.

1. Эндпоинт `POST /api/v1/webhooks/payment` (имя провайдера в пути или заголовке — по их доке).
2. Проверка **подписи** (HMAC / JWT — строго по документации платёжки).
3. **Идемпотентность:** таблица `webhook_events` с уникальным ключом `(provider, event_id)` или хеш тела; повтор — 200 без повторного провижнинга.
4. После валидации: обновить заказ → `paid` → поставить задачу **provision** в очередь.
5. Ответ webhook **быстрый** (< 2–5 с): тяжёлое в worker.

**Критерий готовности:** тест «дважды отправили один webhook» — одна активация провижнинга.

## 9. Фаза 4 — Partner API провайдеров и провижнинг

**Цель:** маркетплейс говорит с провайдерами на одном языке.

### 9.1 Документ для партнёров (отдельный файл позже)

- `POST /partner/v1/subscription/create`
- `GET /partner/v1/subscription/{id}`
- `DELETE /partner/v1/subscription/{id}`
- `GET /partner/v1/health`
- Заголовок **`X-Partner-Token`**, таймауты, коды ошибок (`tariff_not_found`, `capacity_full`, …).

### 9.2 Поведение провижнинга (worker)

1. Взять задачу для `order_id`.
2. Загрузить из БД: `provider.partner_api_url`, расшифрованный токен, маппинг `tariff_id`.
3. `POST create` с телом: `order_id`, `tariff_id`, `email`, `protocol`, `comment`.
4. Успех: сохранить `subscription_id`, **зашифрованный** или сырой JSON конфига (политика хранения — в ADR), перевести заказ в `active`, отправить письмо.
5. Ошибка: `failed` + `error_code`, уведомление оператору, письмо пользователю «оплата получена, разбираемся».

### 9.3 Ретраи

- Сетевые ошибки и 5xx: exponential backoff, максимум N попыток.
- 4xx от провайдера (кроме конфликта): обычно не ретраить без смены входных данных.

**Критерий готовности:** интеграционный тест с **mock Partner API** (WireMock / Prism).

## 10. Фаза 5 — адаптеры (3XUI / Marzban / прочее)

**Цель:** провайдер без своего API поднимает контейнер с вашим адаптером.

**Критерий готовности:** один реальный провайдер проходит тестовую покупку через адаптер.

## 11. Фаза 6 — наблюдаемость, админка, ручные операции

1. **Структурированные логи** (JSON): `request_id`, `order_id`, `provider_id`, длительность вызовов Partner API (частично уже есть в [backend/vps_api/lib/logging.mjs](../backend/vps_api/lib/logging.mjs) — расширять по мере фич).

4. **Алерты** при росте `failed` или отсутствии успешных health провайдеров.

## 12. Данные: схемы сущностей (черновик)

### 12.1 Provider (интеграция — расширение к каталогу)

Поля могут жить в Firestore в документе провайдера **только для публичных** данных; секреты — **только на сервере** в отдельной таблице `provider_credentials`:

- `provider_id`
- `partner_api_url`
- `partner_token_ciphertext`

- `config_response` (зашифровано или ссылка в Secret/Object Storage)
- `error_code`, `error_message`
- `created_at`, `updated_at`, `completed_at`

### 12.3 WebhookEvent (идемпотентность)

- `id`, `source`, `external_event_id`, `payload_hash`, `processed_at`

## 13. Безопасность (чеклист)

- [ ] Секреты не в репозитории и не в клиенте; ротация ключей описана.
- [ ] Webhook только с проверкой подписи; сырой body для HMAC.
- [ ] Rate limit на публичные эндпоинты (КП, создание заказа) по IP + по user.
- [ ] Partner Token на стороне маркетплейса хранить шифрованием (KMS / libsodium).
- [ ] Выдача конфига пользователю — по HTTPS; в письме — осознанный риск (ссылки с ограниченным сроком).
- [ ] Минимальные права сервис-аккаунта Firebase Admin на сервере.

## 14. Производительность и «лёгкость» системы

- **Webhook:** не блокировать на I/O к почте и провайдеру — очередь.
- **Partner API:** connection pooling, таймаут 10–30 с по политике, circuit breaker при массовых ошибках одного провайдера.
- **БД:** индексы по `orders(status, created_at)`, `orders(provider_id)`.
- **Кэш:** публичный каталог не дублировать тяжёлым бэком — клиент уже грузит через VPS; бэк только для транзакций.

## 16. Ревью плана с позиции тестировщика

Проверка по заданным критериям: **понятность**, **удобство**, **скорость**, **лёгкость**.

### 16.1 Понятность

| Аспект | Оценка | Замечания |
|--------|--------|-----------|
| Структура | Хорошо | Фазы идут от безопасности к сложным интеграциям; новый человек видит порядок работ. |
| Риск путаницы | Средне | Нужно явно назвать **один** «источник правды» для OpenAPI (путь в репо), иначе план + код разъедутся. |
| Глоссарий | Не хватает | Добавить мини-словарь: «заказ», «провижнинг», «партнёр», «адаптер», «webhook event id». |

**Рекомендация:** после первой реализации OpenAPI — ссылка на него в первом абзаце этого документа (см. вводный абзац выше).

### 16.2 Удобство сопровождения (для разработчиков и операторов)

| Аспект | Оценка | Замечания |
|--------|--------|-----------|
| Онбординг | Хорошо | Фаза 0 снимает главный анти-паттерн (секреты в приложении). |
| Отладка | Средне | Без `request_id` в логах и ответах поддержка будет страдать — в плане заложено, нужно не забыть при реализации. |
| Ручные сценарии | Ок | Фаза 6 явно про повтор провижнинга и `failed`. |

**Рекомендация:** один файл **RUNBOOK.md** (куда смотреть логи, как повторить webhook вручную на staging).

### 16.3 Скорость (производительность и время отклика)

| Аспект | Оценка | Замечания |
|--------|--------|-----------|
| Горячий путь | Хорошо | Разделение: webhook быстрый, провижнинг асинхронный. |
| Риск | Средне | Если очередь на первом этапе «таблица в БД», при пике возможны блокировки — заранее решить, какой worker её забирает (один процесс или горизонтальное масштабирование с `SELECT FOR UPDATE SKIP LOCKED`). |

**Рекомендация:** для MVP допустима одна нода worker; зафиксировать лимит RPS в нагрузочном чеклисте перед продом.

### 16.4 Лёгкость (не перегружен ли план)

| Аспект | Оценка | Замечания |
|--------|--------|-----------|
| Объём | Высокий | Документ большой — это нормально для мастер-плана, но **исполнителю** нужен **одностраничный** «MVP scope»: фазы 0–3 + mock Partner. |
| Риск переусложнения | Средне | Фазы 5–6 можно отложить без блокировки MVP, если mock/один кастомный партнёр согласован. |

**Рекомендация:** завести в конце документа таблицу **MVP vs Later** или отдельный `docs/BACKEND_MVP.md`.

### 16.5 Итог ревью

План **согласованный и реализуемый**: порядок фаз снижает риск, ключевые нефункциональные требования (секреты, идемпотентность, асинхронный провижнинг) учтены. Для максимальной «понятности и лёгкости» на практике стоит добавить: **глоссарий**, **ссылку на OpenAPI**, **RUNBOOK**, **краткий MVP-вырез** — тогда тестировщик и новый бэкенд-разработчик закроют сценарии без расспросов.

## Глоссарий

| Термин | Значение |
|--------|----------|
| **Заказ (Order)** | Запись о намерении покупки / оплате / выдаче подписки со статусной машиной. |
| **Провижнинг** | Асинхронный вызов Partner API (или адаптера) для создания подписки у провайдера после оплаты. |
| **Партнёр** | VPN-провайдер с API или адаптером, интегрируемый маркетплейсом. |
| **Адаптер** | Прослойка (часто Docker), переводящая единый Partner API в вызовы панели (3x-ui, Marzban и т.д.). |
| **Webhook event id** | Внешний идемпотентный ключ события от платёжки; повтор доставки не должен дублировать побочные эффекты. |
| **VPS API** | Текущий Node-процесс в `backend/vps_api` — прокси Firestore + Resend + эндпоинты `/api/v1/…`. |
| **Полный прокси** | Каталог, отзывы, чат, sync к данным Firestore только через VPS при закрытых client rules. |
| **Bearer** | Firebase ID token в заголовке `Authorization` для защищённых эндпоинтов. |
