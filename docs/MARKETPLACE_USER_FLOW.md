# Сквозной сценарий: пользователь в приложении → оплата → подписка у провайдера

Документ описывает **полный путь** для конечного пользователя и для систем: приложение, VPS API маркетплейса (Firestore), воркер провижнинга, сервер провайдера с типовым стеком «VPN Bot» (`/root/bot`) и патчем [partner-patches/vpn-bot-stack/README.md](../partner-patches/vpn-bot-stack/README.md).

**Патч у провайдера — не одна команда «запустил и всё»:** нужно скопировать файлы на сервер, `pip install` в venv бота, создать `.env.partner` (`PARTNER_TOKEN`, `TARIFF_MAP`), поставить systemd-unit, прокинуть `location /partner/v1/` в nginx и прописать `partnerApiUrl` / `partnerApiToken` в Firestore. Подробно — в README патча.

---

## 1. Что видит пользователь в приложении

1. Открывает каталог — приложение обращается к **VPS API маркетплейса** (`GET /api/v1/catalog` и др.); данные каталога приходят из **Firestore** только через этот бэкенд (клиентский SDK к Firestore для каталога не используется).
2. Выбирает **провайдера** и **тариф** (у каждого провайдера свой `providerId` и свои `tariff_id` в подколлекции тарифов).
3. Вводит **email** (и при необходимости комментарий / протокол), инициирует оплату.
4. Приложение создаёт заказ: `POST /api/v1/orders` → в ответе `order_id`, статус например `pending_payment`.
5. Пользователь **оплачивает** (тестовый webhook или реальная платёжка — принцип один: успех привязывается к этому `order_id`).
6. После подтверждения оплаты бэкенд маркетплейса переводит заказ в **`paid`**, воркер провижнинга забирает заказ и вызывает **Partner API** на стороне провайдера.
7. В приложении при опросе `GET /api/v1/orders/{id}` пользователь видит `provisioning` → **`active`** (или ошибку).
8. На указанный **email** уходит письмо маркетплейса (Resend): **ссылка на подписку** и/или данные из поля `config` в ответе провайдера. Для Remnawave чаще приходит **URL подписки** (импорт в клиент), а не один `vless://` в JSON — это определяется панелью и тем, что вернула цепочка `grant_subscription`.

Итог для пользователя: одно приложение, одна оплата через маркетплейс, одно письмо с инструкцией подключения — без обязательного захода в Telegram-бота провайдера (если так задуман продукт).

## 2. Поток по шагам (кто куда стучится)

| Роль | Куда ходит |
|------|------------|
| Приложение (Flutter) | Только **VPS API маркетплейса** (HTTPS). |
| VPS API | **Firestore** (Admin SDK), **Resend**; воркер — **HTTPS провайдера** (Partner API). |
| Воркер провижнинга | **HTTPS провайдера** `POST .../partner/v1/subscription/create` с `X-Partner-Token`. |
| Патч на сервере провайдера | Внутри: **`db_helpers`** + **`grant_subscription`** → **SQLite `vpn_bot.db`**, **Remnawave / X-UI** (как после оплаты в TG-боте). |

### A. Каталог и тарифы

- **Источник правды:** Firestore (`providers/{providerId}`, тарифы, агрегаты отзывов и т.д.).
- **Клиенту отдаёт:** только маркетплейс VPS API.
- **Провайдер** на этом шаге **не вызывается**.

### B. Создание заказа

- Приложение → `POST /api/v1/orders` (`providerId`, `tariffId`, `email`, …).
- Маркетплейс пишет **Firestore** `orders/{orderId}` (например `pending_payment`, привязка к `userId` Firebase и т.д.).
- Провайдер **ещё не участвует**.

### C. Оплата

- Платёжка шлёт webhook на **маркетплейс**.
- Маркетплейс обновляет заказ (**`paid`**, очередь провижнинга).
- Провайдер **пока не вызывается**.

### D. Провижнинг (маркетплейс → провайдер)

- Воркер читает заказ и **`provider_credentials/{providerId}`**: `partnerApiUrl`, `partnerApiToken`, при необходимости `timeoutMs`.
- Запрос: **`POST {partnerApiUrl}/partner/v1/subscription/create`** с заголовком **`X-Partner-Token`** и телом: `order_id`, `tariff_id` (тот же id, что в каталоге), `email`, опционально `protocol`, `comment`.

### E. У провайдера (патч + логика бота)

1. Проверка токена.
2. Разбор **`tariff_id`** по **`TARIFF_MAP`** в `.env.partner` на сервере провайдера (дни, ГБ, `limit_ip` — должны соответствовать показанному в каталоге тарифу).
3. Стабильный **синтетический отрицательный `telegram_id`** от `order_id` (не пересекается с реальными Telegram ID).
4. При отсутствии пользователя в **`vpn_bot.db`** — **`add_user`** (username из email, `real_username` = email).
5. **`grant_subscription`** — та же цепочка, что после успешной оплаты в Telegram-боте: Remnawave и/или X-UI, мульти-сервер, лимиты, запись в БД провайдера. Сквады и дефолты — из **настроек бота** (`app_conf`), как при оплате через бота.
6. Ответ маркетплейсу в формате **Partner API**: `subscription_id`, по возможности `subscription_url`, `expires_at`, `config`.

### F. Обратно пользователю

- Воркер обновляет **Firestore** (заказ **`active`**, поля подписки, ссылки).
- **Resend** отправляет письмо на email из заказа.
- Приложение при опросе заказа показывает успех (и при необходимости ссылку в UI).

## 3. Где что хранится

| Место | Данные |
|--------|--------|
| **Firestore (маркетплейс)** | Каталог, заказы, `provider_credentials`, пользовательские sync-данные приложения и т.д. |
| **VPS API + воркер** | Оркестрация; долгосрочная «истина» о подписке на панели провайдера — у провайдера. |
| **Сервер провайдера** | **`vpn_bot.db`**: пользователь с синтетическим `telegram_id`, подписка, UUID в той же модели, что у бота. **Панель** (Remnawave/X-UI): реальный клиент. |
| **`.env.partner` у провайдера** | `PARTNER_TOKEN`, `TARIFF_MAP` — операционный конфиг **на сервере провайдера**, не в Firestore маркетплейса. |

---

## 4. Что видит провайдер

- **Входящие:** HTTPS с маркетплейса на `/partner/v1/...`, заголовок **`X-Partner-Token`**, тело с `order_id`, `tariff_id`, `email`.
- **В БД и панели:** записи как при продаже через бота; в логах — та же `grant_subscription`.
- **Исходящие «в маркетплейс»:** только **HTTP-ответ** на `create` / `delete`. Статус заказа в приложении обновляет **маркетплейс** после ответа воркера (провайдер сам push не шлёт).

---

## 5. Много провайдеров с однотипным беком

- В Firestore у каждого свой **`providerId`**, свои тарифы в каталоге, своя запись **`provider_credentials`** (свой `partnerApiUrl` и токен).
- На каждом сервере — **тот же патч** из `partner-patches/vpn-bot-stack/` и **свой `TARIFF_MAP`**, согласованный с id тарифов в каталоге.
- Приложение всегда работает с **одним** API маркетплейса; различие провайдеров — данные каталога и URL, куда воркер отправляет `POST`.
