# Онбординг VPN-провайдера в маркетплейс

## 1. Как это работает в двух фразах

1. Пользователь покупает тариф в приложении; оплата проходит через интеграцию маркетплейса.
2. После подтверждения оплаты **наш сервер** отправляет на **ваш HTTPS-эндпоинт** запрос «создать подписку» с `order_id`, `tariff_id`, `email`. Вы создаёте кл/иента в своей системе и возвращаете `subscription_id` (и по возможности ссылку/срок действия/конфиг).

Ваши панельные логины и внутренние API **не торчат наружу** в приложение: с приложением общается только маркетплейс, к вам ходит только наш бэкенд.

## 2. Что нужно подготовить до разработки

### 2.1. Идентификатор провайдера (`providerId`)

Согласуйте с оператором маркетплейса строковый ID (латиница, без пробелов), например `acme_vpn`. Он же используется в каталоге: документ `providers/{providerId}` и подколлекция тарифов `providers/{providerId}/tariffs/{tariff_id}`.

### 2.2. Согласование тарифов (`tariff_id`)

Для **каждого** тарифа, который виден в приложении, ID в Firestore (`tariffs` subcollection) **должен совпадать** с тем значением `tariff_id`, которое ваш сервер принимает в `POST /partner/v1/subscription/create`.

Пример: в каталоге документ `providers/acme_vpn/tariffs/pro_month` → в запросе придёт `"tariff_id": "pro_month"`. Если у вас в панели другие имена — маппинг делаете **у себя** (в своём сервисе или адаптере).

### 2.3. Публичный HTTPS URL (`partnerApiUrl`)

- Схема **HTTPS** (валидный сертификат).
- Без обязательного завершающего `/` в конце базового URL (маркетплейс сам склеивает пути `/partner/v1/...`).
- Доступен **исходящим** запросам с серверов маркетплейса (firewall / allowlist по IP — обсудите с оператором, если нужен whitelist).

### 2.4. Секрет (`partnerApiToken`)

Длинная случайная строка (например 32+ байт в hex или аналог `sk_live_...`). Передаётся в заголовке **`X-Partner-Token`** на каждый запрос. Храните только на сервере; в мобильное приложение не кладётся.

Оператор маркетплейса сохранит пару `partnerApiUrl` + `partnerApiToken` в Firestore в документе **`provider_credentials/{providerId}`** (пишет только сервер маркетплейса).

## 3. Что реализовать на вашей стороне (минимум)

| Метод | Путь | Обязательность |
|--------|------|------------------|
| `GET` | `/partner/v1/health` | **Да** (быстрый ответ, тот же `X-Partner-Token`) |
| `POST` | `/partner/v1/subscription/create` | **Да** |
| `GET` | `/partner/v1/subscription/{subscription_id}` | Опционально (MVP может не вызывать) |
| `DELETE` | `/partner/v1/subscription/{subscription_id}` | Опционально |

Точные схемы JSON, коды ошибок и примеры — в **[openapi/partner-api.yaml](openapi/partner-api.yaml)** (откройте в [Swagger Editor](https://editor.swagger.io)) и в [PARTNER_API.md](PARTNER_API.md).

### 3.1. Успешный ответ на `subscription/create`

HTTP **200**, тело JSON. Обязательно вернуть идентификатор подписки в одном из видов:

- `subscription_id`, или
- legacy: поле `id`

Желательно также:

- `subscription_url` — ссылка для импорта в клиент VPN;
- `expires_at` — ISO 8601;
- `config` — объект с параметрами подключения (для письма пользователю и отображения).

### 3.2. Бизнес-ошибка (тариф не найден, нет места и т.д.)

Допустимо:

- HTTP **200** с `"status": "error"`, `error_code`, `message`, **или**
- HTTP 4xx/5xx.

Рекомендуемые `error_code`: `tariff_not_found`, `capacity_full`, `invalid_request`, `unauthorized`.

### 3.3. Таймауты

Маркетплейс использует таймаут порядка **10–30 секунд** (настраивается полем `timeoutMs` в `provider_credentials`). Ваш обработчик не должен блокироваться на долгие синхронные операции без очереди.

## 4. Если есть только панель (3x-ui, Marzban, …), без своего HTTP API

Нужен **отдельный небольшой сервис** (контейнер или процесс) **на стороне провайдера**, который:

1. Реализует пути `/partner/v1/*` по контракту.
2. Внутри вызывает API панели / БД с вашими админскими учётными данными (они не передаются маркетплейсу).

В репозитории маркетплейса есть заготовка [partner-adapter/README.md](../partner-adapter/README.md) (каркас; логику создания подписки под вашу панель дописываете вы или по договорённости с оператором).

## 5. Передача данных оператору маркетплейса

После того как staging/production URL готовы, передайте оператору (безопасным каналом):

1. **`providerId`**
2. **`partnerApiUrl`** (production)
3. **`partnerApiToken`**
4. Таблицу соответствия: *ID тарифа в каталоге* → *что создаётся у вас* (если не очевидно из названий)

Оператор создаёт/обновляет документ `provider_credentials/{providerId}` и при необходимости выключает тестовый mock.

## 6. Самопроверка (curl)

Подставьте свой базовый URL и токен.

**Health**

```bash
curl -sS -H "X-Partner-Token: ВАШ_СЕКРЕТ" "https://api.ваш-домен/partner/v1/health"
```

**Создание подписки (тест)**

```bash
curl -sS -X POST "https://api.ваш-домен/partner/v1/subscription/create" \
  -H "X-Partner-Token: ВАШ_СЕКРЕТ" \
  -H "Content-Type: application/json" \
  -d '{"order_id":"test_order_1","tariff_id":"СОГЛАСОВАННЫЙ_ID","email":"test@example.com","protocol":"vless"}'
```

Ожидается HTTP 200 и JSON с `subscription_id` или `id` при успехе.

Референсный mock для локальной отладки у оператора: `npm run mock-partner` в каталоге `backend/vps_api` (см. [PARTNER_API.md](PARTNER_API.md)).

## 7. Чеклист перед продакшеном

- [ ] HTTPS с валидным сертификатом
- [ ] `GET /partner/v1/health` с `X-Partner-Token` возвращает 200
- [ ] Все тарифы из каталога имеют согласованные `tariff_id` и обрабатываются в `subscription/create`
- [ ] Идемпотентность: повторный запрос с тем же `order_id` либо безопасен, либо возвращает тот же результат (маркетплейс может ретраить при сетевых сбоях)
- [ ] Секрет только на сервере; ротация токена согласована с оператором
- [ ] Логи без полного содержимого секретов и без лишних ПДн

## 8. Вопросы не по контракту

Интеграция оплаты, возвраты, SLA, юридические условия — отдельно с оператором маркетплейса. Этот документ описывает только **технический** интерфейс Partner API.
